# 이벤트 이미지 전면 교체 설계

## 목적

- Firestore `events` 컬렉션의 이벤트 22개 이미지를 실제 종합 패션몰처럼 서로 다른 캠페인 비주얼로 교체한다.
- 여성·남성·혼성 그룹·제품 단독 컷을 이벤트별로 배분한다.
- 행사명과 핵심 혜택 한글 문구를 이미지 생성 단계에서 직접 포함한다.
- 기존 이벤트 목록·상세·관리자 기능과 Firebase 이미지 필드 구조를 유지한다.

## 범위

- 대상: Firestore `events` 컬렉션의 이벤트 22개 전체
- 생성 수량: 이벤트당 와이드 1개와 카드 1개, 총 44개
- 코드 범위:
  - 이벤트 목록의 중복 텍스트 오버레이 축소
  - 모바일 대표·상세 영역의 카드 이미지 사용
  - 이미지 역할과 반응형 표시를 검증하는 테스트
- 데이터 범위:
  - Firebase Storage 새 버전 파일 업로드
  - Firestore 이벤트 문서의 `bannerImage`, `thumbnailImage`, `detailImage` 전환
- 제외:
  - 기존 Storage 파일 삭제
  - 기존 `public/events/2026` 이미지 삭제
  - 이벤트 제목·기간·혜택·참여 정책 변경
  - 배포, 커밋, 푸시

## 이미지 규격

### 와이드 포스터

- 최종 규격: `1600x820` WebP
- 용도: `bannerImage`, `detailImage`
- 구성: 왼쪽 한글 카피, 오른쪽 인물·상품 중심
- 포함 문구: 행사명 1줄과 핵심 혜택 1줄
- 제외 문구: 기간, 참여자 수, 쿠폰 코드, CTA

### 카드 포스터

- 최종 규격: `1000x1250` WebP
- 용도: `thumbnailImage`, 모바일 대표·상세 이미지
- 구성: 세로 비율에 맞는 독립 구도
- 포함 문구: 행사명과 축약된 핵심 혜택
- 와이드 이미지를 단순 크롭해서 만들지 않고 별도로 생성한다.

### 공통 제작 규칙

- 한글 문구는 이미지 생성 단계에서 직접 생성한다. 별도 텍스트 합성으로 교정하지 않는다.
- 글자 오류, 잘림, 의미 변경이 있으면 해당 이미지를 다시 생성한다.
- 가짜 브랜드 로고, 워터마크, 불필요한 영문 문구를 포함하지 않는다.
- 동일한 인물, 포즈, 촬영 공간, 색상 조합이 연속해서 반복되지 않게 한다.
- 둥근 그래픽 스티커와 장식성 그림자를 피하고 사진과 큰 한글 타이포를 중심으로 구성한다.
- 파일 크기는 Firebase Storage 규칙에 맞게 5MB 미만으로 유지한다.

## 화면 적용

### 이벤트 목록

- 대표 히어로는 데스크톱에서 `detailImage`, 모바일에서 `thumbnailImage`를 사용한다.
- 이미지 안의 행사명·혜택과 중복되는 UI 제목·설명은 시각적으로 제거한다.
- 상태, 기간, CTA, 진행 이벤트 통계는 UI 텍스트로 유지한다.
- 접근성 이름에는 이벤트 제목과 설명을 유지한다.
- 목록 카드는 `thumbnailImage`를 사용한다.
- 카드 오버레이는 유형·상태·기간·CTA만 표시하고 제목·설명·혜택 반복은 제거한다.

### 이벤트 상세

- 상단 이벤트 제목, 설명, 기간, 참여 CTA는 접근성과 정보 구조를 위해 유지한다.
- 배너 영역은 데스크톱에서 `detailImage`, 모바일에서 `thumbnailImage`를 사용한다.
- 우측 혜택 요약 패널과 상세 정보 섹션은 유지한다.

## Firebase 저장 구조

- 프롬프트 매니페스트: `scripts/event-image-refresh-manifest.json`
- 프로젝트 와이드 파일: `public/events/2026-v2/{eventId}-wide.webp`
- 프로젝트 카드 파일: `public/events/2026-v2/{eventId}-card.webp`
- 와이드 파일: `events/banner/{eventId}-20260714-wide.webp`
- 카드 파일: `events/thumbnail/{eventId}-20260714-card.webp`
- Firestore 필드:
  - `bannerImage`: 와이드 파일 URL
  - `detailImage`: `bannerImage`와 동일한 와이드 파일 URL
  - `thumbnailImage`: 카드 파일 URL
- 기존 Storage 객체는 삭제하지 않는다.
- 관리자 화면의 기존 `events/banner`, `events/thumbnail`, `events/detail` 업로드 흐름은 변경하지 않는다.

## 제작 및 전환 흐름

1. 이벤트 ID, 정확한 한글 문구, 와이드·카드 프롬프트를 매니페스트로 관리한다.
2. 이미지 생성 기능을 사용해 44개 이미지를 한 장씩 생성한다.
3. 각 생성물의 한글, 구도, 인물·상품 형태를 즉시 확인한다.
4. 오류가 있는 생성물은 후처리 교정 없이 다시 생성한다.
5. 승인된 이미지를 최종 규격의 WebP로 변환해 `public/events/2026-v2`에 보관한다.
6. 와이드와 카드 컨택트시트를 만들어 전체 22개 사이의 반복성을 점검한다.
7. Firebase 전환 전 현재 22개 이벤트의 이미지 필드를 저장소 밖의 임시 백업에 보관한다.
8. 새 이미지 44개를 버전명 Storage 경로에 업로드한다.
9. 업로드된 44개 URL과 이미지 응답을 전부 확인한다.
10. 모든 업로드가 성공한 경우에만 Firestore batch write로 22개 이벤트 문서를 전환한다.
11. 목록·상세·모바일 표시와 Firestore 필드 값을 검증한다.

## 실패 및 복구 정책

- 이미지 생성 실패: 해당 이벤트만 재생성하고 다음 이벤트로 넘어가지 않는다.
- 한글 오류: 별도 합성으로 수정하지 않고 재생성한다.
- WebP 변환 또는 크기 검증 실패: Firebase에 업로드하지 않는다.
- 일부 Storage 업로드 실패: Firestore batch를 실행하지 않는다.
- Firestore batch 실패: 기존 문서 값이 유지되는지 확인하고 재실행하지 않은 상태로 원인을 분석한다.
- 전환 후 표시 오류: 임시 백업의 이전 이미지 필드로 22개 문서를 되돌릴 수 있게 한다.
- Firebase 관리자 인증이 없으면 로컬 이미지·검증 완료 지점에서 중단하고 Firestore는 변경하지 않는다.

## 검증

- 이미지 자동 검증:
  - 파일 존재 여부
  - `1600x820`, `1000x1250` 크기
  - WebP 형식
  - 파일당 5MB 미만
- 이미지 수동 검증:
  - 행사명과 혜택 한글의 정확성
  - 문구 잘림 여부
  - 인물 손·얼굴과 상품 형태
  - 이미지 간 인물·구도·배경·색상 반복성
- 코드 검증:
  - 이벤트 목록 컴포넌트 테스트
  - 이벤트 이미지 유틸 테스트
  - 타입체크
  - 경고 없는 린트
  - 전체 Jest 테스트
- Firebase 검증:
  - Storage 44개 객체 접근 가능 여부
  - Firestore 22개 문서의 세 이미지 필드
  - 목록 22개 링크와 상세 22개 URL

## 완료 기준

- 이벤트 22개가 모두 서로 구분되는 와이드·카드 비주얼을 가진다.
- 44개 이미지에 승인된 한글 행사명과 혜택이 정확하게 포함된다.
- 데스크톱과 모바일에서 이미지 속 문구가 잘리지 않는다.
- 이미지와 UI 텍스트가 중복되어 읽기 어려운 상태가 없다.
- Firebase Storage 업로드와 Firestore batch 전환이 전부 성공한다.
- 기존 이벤트 참여·쿠폰·필터·상세 동작이 유지된다.
- 기존 이미지가 삭제되지 않아 이전 상태로 복구할 수 있다.
