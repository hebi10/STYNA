# 이벤트 상세 에디토리얼 스토리 설계

## 배경

현재 이벤트 상세는 대형 캠페인 이미지 아래에 제목, 기간, 혜택 요약, 상품, 긴 안내를 순서대로 배치한다. 상품 카드는 실제 쇼핑 기능과 연결되어 있지만 상단 정보가 반복되고, 상품과 이벤트 이미지 사이에 스타일 제안이나 혜택 설명이 없어 일반 안내 페이지처럼 보인다.

## 결정

Firestore 이벤트 22개에 상세 본문용 이미지 3장씩 총 66장을 새로 제작한다. 기존 가로 이미지는 상단 요약 영역의 작은 썸네일로 유지하고, 신규 이미지는 `혜택/쿠폰 → MD 추천 룩북 → 상품·소재 에디토리얼` 순서로 이어 붙인다. 신규 이미지 다음에 관련 상품 8개와 짧은 유의사항을 배치한다.

## 목표

- 이벤트 상세를 실제 종합 패션몰 기획전처럼 이미지 중심의 구매 흐름으로 바꾼다.
- 이벤트별 신규 이미지 3장의 인물, 계절, 팔레트와 타이포 톤을 한 캠페인처럼 통일한다.
- 여성, 남성, 혼성 그룹, 제품 단독 컷을 22개 이벤트 전체에서 분산한다.
- 이미지 안 행사명, 할인율, 쿠폰, MD 추천 문구는 정확한 한글로 직접 생성한다.
- 기존 이벤트 참여, 로그인, 쿠폰 지급, 상품 조회, Firebase 이미지 동작을 보존한다.
- 신규 이미지가 없거나 일부만 있는 이벤트도 기존 데이터로 안전하게 표시한다.

## 비목표

- 기존 배너와 카드 이미지 객체를 삭제하거나 덮어쓰지 않는다.
- 이벤트 참여 API, 쿠폰 정책, 상품 가격과 재고 계산을 변경하지 않는다.
- 이벤트마다 별도의 상세 컴포넌트를 만들지 않는다.
- 생성 이미지 위에 동일한 행사명과 혜택을 UI 텍스트로 다시 덮지 않는다.

## 화면 구조

### 1. 상단 요약

- 데스크톱은 `기존 가로 이미지 45% / 이벤트 요약 55%`의 2열 구조를 사용한다.
- 기존 `detailImage`, 없으면 `bannerImage`를 작은 가로 이미지로 표시한다.
- 우측에는 상태, 행사명, 기간, 짧은 설명, 핵심 혜택, 주 CTA, `기획전 상품 보기` 앵커를 둔다.
- 기존 별도 제목 블록과 3열 사양표 형태의 행동 바는 제거해 정보 반복을 줄인다.
- 모바일은 이미지와 요약을 세로로 쌓고 기존 하단 고정 CTA를 유지한다.

### 2. 에디토리얼 이미지 스토리

- 신규 세로형 상세 이미지 3장을 한 컨테이너 안에서 순서대로 배치한다.
- 각 이미지는 원본 비율 전체를 표시하고 `object-fit: contain`을 사용해 인물과 한글을 자르지 않는다.
- 이미지 사이에는 간격을 두지 않거나 1px 이하 구분만 사용해 하나의 긴 기획전 본문처럼 이어지게 한다.
- 각 이미지는 단일 배너가 아니라 2~3개의 사진 장면과 짧은 설명 카피를 가진 상세 콘텐츠 조각으로 구성한다.
- 이미지 역할은 다음과 같이 고정한다.
  1. `benefit`: 행사 혜택, 할인율, 쿠폰 금액, 참여 보상을 시각적으로 설명한다.
  2. `styling`: 여성, 남성 또는 혼성 모델을 활용해 이벤트 계절과 상품군에 맞는 MD 스타일링을 제안한다.
  3. `product`: 제품 단독 컷, 소재 클로즈업 또는 착용 디테일로 추천 상품군을 연결한다.

### 3. 상품 매대

- 기존 이벤트 상품 선택 로직과 `ProductCard`를 재사용해 최대 8개를 4열 × 2줄로 표시한다.
- 섹션 제목은 이벤트 변형에 맞춰 유지하되, 상품 카드에 이벤트 대상 라벨과 짧은 MD 추천 문구를 전달한다.
- 오류, 재시도, 빈 상품 상태와 상품 목록 링크는 현재 동작을 보존한다.
- 모바일은 2열을 사용한다.

### 4. 짧은 유의사항

- 상품 아래에는 `혜택·사용 방법`과 `유의사항` 두 영역만 둔다.
- 긴 자동 생성 문장을 줄이고 실제 이벤트 데이터에 있는 혜택, 기간, 대상, 정원, 쿠폰 지급 조건만 표시한다.
- 데스크톱과 모바일 모두 접고 펼칠 수 있으며 첫 영역만 기본으로 연다.
- 하단에는 관련 상품과 전체 이벤트 링크를 유지한다.

## 이미지 제작 규격

- 수량: 22개 이벤트 × 3장 = 66장
- 원본 규격: `1000 × 1500` WebP
- 생성 방식: built-in 이미지 생성 도구를 사용해 이미지별로 한 번씩 생성한다.
- 텍스트: 행사명과 역할별 핵심 한글 문구를 이미지 안에 직접 포함한다.
- 모바일 안전 영역: 핵심 인물과 한글은 가로 중앙 88% 안에 배치하고, 축소 시 읽히도록 짧고 큰 문구를 사용한다.
- 한 장의 최대 정보량: 사진 장면 3개 이하, 제목 포함 한글 카피 4개 이하로 제한한다.
- 금지: 워터마크, 타사 브랜드 로고, 의미 없는 영문, 깨진 한글, UI 버튼처럼 보이는 가짜 컨트롤, 과도한 소품 반복
- 파일명: `{eventId}-20260715-{benefit|styling|product}.webp`

## 이벤트별 시각 일관성

- 한 이벤트의 3장은 같은 계절, 팔레트, 조명과 스타일 방향을 공유한다.
- 동일 인물의 얼굴을 억지로 반복하는 것보다 캠페인 캐스팅과 의상 톤을 일치시킨다.
- 22개 이벤트 사이에서는 다음 요소를 분산한다.
  - 여성 단독, 남성 단독, 2인 혼성, 3인 혼성, 제품 단독
  - 스튜디오, 도심, 리조트, 해변, 오피스, 매장, 자연광 실내
  - 전신, 반신, 움직임이 있는 장면, 플랫레이, 소재 클로즈업
- 각 이벤트의 3장 모두에 큰 제목을 반복하지 않는다. 첫 장은 혜택, 둘째 장은 MD 추천, 셋째 장은 상품군 연결에 집중한다.

## 데이터와 Firebase 계약

`Event`에 선택적 `editorialImages` 객체를 추가한다.

```ts
editorialImages?: {
  benefit?: string;
  styling?: string;
  product?: string;
};
```

- 기존 `bannerImage`, `thumbnailImage`, `detailImage`는 유지한다.
- Storage에는 `events/editorial/{eventId}-20260715-{role}.webp` 경로로 신규 객체만 업로드한다.
- Firestore 22개 문서는 세 역할의 URL 검증이 모두 끝난 뒤 한 batch에서 `editorialImages`를 추가한다.
- 전환 전 이미지 관련 필드는 로컬 백업 JSON으로 저장한다.
- 업로드 스크립트는 프로젝트와 버킷 일치, 기존 객체 미존재, WebP 형식, `1000 × 1500`, 파일 수 66개를 먼저 확인한다.
- 관리자 이벤트 폼에는 세 역할을 구분한 선택적 업로드 슬롯을 추가한다.

## 폴백과 오류 처리

- `editorialImages`가 없으면 에디토리얼 스토리를 렌더링하지 않고 기존 상품 섹션으로 이어진다.
- 세 역할 중 일부가 없으면 존재하는 이미지만 역할 순서대로 표시한다.
- 이미지 로딩 실패는 해당 이미지만 감추고 이벤트 요약, CTA와 상품 매대는 유지한다.
- Firebase 업데이트 실패 시 신규 Storage 객체는 삭제하지 않고 Firestore는 기존 필드를 유지한다.
- 롤백은 백업 JSON의 기존 이미지 필드와 `editorialImages` 이전 상태를 복원한다.

## 컴포넌트 구조

- `EventDetailClient`: 상태, 참여, CTA와 전체 섹션 순서를 소유한다.
- `EventCommerceHero`: 작은 기존 이미지와 이벤트 요약을 2열로 렌더링한다.
- `EventEditorialStory`: 신규 이미지 3장의 역할 순서, 폴백과 접근 가능한 대체 텍스트를 담당한다.
- `EventProductShowcase`: 기존 상품 로더를 유지하고 이벤트 대상/MD 문맥을 상품 카드에 전달한다.
- `EventInformationSections`: 두 개의 짧은 아코디언으로 축약한다.

## 테스트와 검수

### 자동 검증

- 이미지 manifest가 22개 이벤트와 역할 3개를 정확히 포함하는지 검사한다.
- 66개 파일의 WebP 형식, `1000 × 1500`, 파일명과 이벤트 ID를 검사한다.
- Storage 업로드 전 생성 전용 조건과 Firestore batch 입력을 테스트한다.
- 신규 필드가 없는 기존 이벤트의 폴백을 컴포넌트 테스트로 검증한다.
- 3장 순서, 일부 누락, 전체 누락, 대체 텍스트와 상품 섹션 DOM 순서를 검증한다.
- 기존 참여/로그인/종료 CTA, 상품 로딩·오류·재시도 테스트를 유지한다.

### 시각 검수

- 66장을 한 장씩 확인해 한글 행사명과 혜택 문구가 정확한지 검사한다.
- 인물 손가락, 얼굴, 의상 구조, 제품 형태, 중복 인물과 배경 반복을 확인한다.
- 같은 이벤트의 3장이 하나의 캠페인으로 보이는지 확인한다.
- 대표 변형 5개와 최종 22개 상세를 데스크톱과 모바일에서 확인한다.
- 이미지 잘림, 한글 가독성, 상품 시작 위치, 고정 CTA 겹침, 가로 넘침과 콘솔 오류를 확인한다.

## 완료 기준

- Firestore 이벤트 22개 모두에 검증된 신규 이미지 3개 URL이 연결된다.
- 사용자 화면에서 기존 이미지 요약 다음에 신규 이미지 3장, 상품 8개, 짧은 유의사항이 순서대로 표시된다.
- 이미지 안 한글이 깨지거나 UI 텍스트와 중복되지 않는다.
- 기존 Firebase 이벤트 CRUD, 참여, 쿠폰과 상품 이동 동작이 회귀하지 않는다.
- 전체 타입 검사, 린트, 테스트, Functions 빌드와 Next.js 빌드가 통과한다.
