# 이미지 전송 성능 개선 설계

## 목표

메인과 카테고리 화면의 이미지 전송량과 재방문 대기 시간을 줄이되, 기존 Firebase Storage 경로와 상품 링크 동작은 유지한다.

## 확인된 원인

- 카테고리 4개 PNG는 총 약 6MB이며, 화면에서 직접 사용된다.
- Next 이미지 최적화가 전역 비활성화되어 Storage 원본을 그대로 요청한다.
- 메인 배너 Storage 응답은 `private, max-age=0`이며, 캐러셀은 복제 슬라이드를 포함해 이미지 요소 14개를 만든다.

## 결정

1. 카테고리 이미지는 우선 로컬 WebP q75로 전환한다. Firebase Storage 업로드는 Admin 자격 증명이 있는 환경에서만 실행하며, 기존 PNG는 삭제하지 않는다.
2. Next 이미지 최적화는 배포 환경에서 활성화하고, 개발 환경만 원본 요청을 유지한다.
3. 업로드 및 마이그레이션 대상 이미지에는 버전 파일명과 함께 장기 브라우저 캐시 메타데이터를 설정한다.
4. 메인 배너는 현재·이전·다음 슬라이드만 이미지 요소를 렌더링한다. 첫 화면의 양쪽 카드만 우선 로드한다.

## 검증 기준

- 카테고리 기본 URL이 더 이상 PNG가 아니다.
- 배너 첫 화면 두 장은 우선 로드되고, 먼 슬라이드의 이미지는 DOM에 없다.
- 이미지 업로드/마이그레이션 메타데이터와 관련 단위 테스트가 통과한다.
- typecheck, lint, 테스트, Functions 빌드와 Next 빌드가 통과한다.
