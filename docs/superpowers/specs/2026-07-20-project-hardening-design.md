# 프로젝트 정리·안정화 설계

## 배경

STYNA는 상품 탐색, 주문 트랜잭션, 구매 인증 리뷰, 이벤트, 관리자 기능을 포함한 커머스 포트폴리오다. 현재 구현 범위와 시각 완성도는 높지만, 미완성 사용자 기능과 중복 관리 화면, 계정·권한 신뢰 경계, 안내 정책과 실제 동작의 불일치, 접근성 및 전역 데이터 조회 구조에 개선이 필요하다.

이번 작업은 사용자에게 노출되는 기능의 신뢰도를 높이고, 운영상 필요한 비공개 코드는 보존하면서 명백한 데드·중복 코드만 제거하는 것을 목표로 한다.

## 목표

1. 참조가 끊긴 코드, 중복 구현, 가치가 없는 빈 기능을 근거 기반으로 제거한다.
2. 계정 상태, 관리자 권한, Firestore 직접 쓰기, 공개 AI 상담의 신뢰 경계를 강화한다.
3. 혜택·결제·배송·이벤트 안내를 실제 서버 동작과 일치시킨다.
4. 로그인 전환과 주문 완료 후에도 사용자의 구매 의도와 최신 데이터를 보존한다.
5. 구매 핵심 화면의 접근성과 모바일 탐색성을 개선한다.
6. 전역 Firestore 중복 조회를 줄이고 페이지별 SEO 정보를 정확히 제공한다.
7. 정적 검사, 단위 테스트, 규칙 테스트, 프로덕션 빌드와 브라우저 QA가 재현 가능한 상태를 만든다.

## 비범위

- 실제 PG 결제 연동
- 이메일 발송 또는 관리자 초대 시스템 신규 구축
- 재입고 알림 백엔드 신규 구축
- 별도 상담 티켓 시스템 신규 구축
- 요청과 무관한 디자인 전면 개편
- 기존 데이터의 파괴적 초기화 또는 대량 마이그레이션
- 커밋, 푸시, 배포

## 공통 제약

- 기존 사용자 변경인 `docs/image-delivery-performance.md`, `next.config.test.ts`, `next.config.ts`를 보존한다.
- 관리자, API, Functions, 데이터 관리 스크립트는 사이트에 직접 노출되지 않는다는 이유만으로 제거하지 않는다.
- 삭제 전 import, 링크, 라우팅, 서비스 호출, 테스트와 문서 참조를 확인한다.
- 기존 스타일과 프레임워크를 우선하며 그림자와 라운드를 추가하지 않는다.
- 기능 변경은 실패 테스트를 먼저 작성하고 최소 구현으로 통과시킨다.
- 동작, 구조, 명령 또는 정책이 바뀐 경우 관련 문서를 짧게 갱신한다.

## 단계 1: 데드·중복 코드 정리

### 삭제 판정 기준

다음 조건을 모두 만족할 때만 코드를 삭제한다.

1. 런타임 import, 동적 import, 라우트, 링크, 테스트 또는 스크립트에서 참조되지 않는다.
2. 관리자·API·Functions·마이그레이션처럼 의도적으로 비공개인 기능이 아니다.
3. 후속 단계에서 연결할 가치 있는 기능이 아니다.
4. 삭제 후 타입체크, 관련 테스트와 프로덕션 빌드가 통과한다.

### 사용자 기능 정리

- `/mypage/counsel`은 별도 데이터 모델이 없으므로 `/mypage/qa`의 문의 내역으로 통합하고 중복 메뉴와 라우트를 제거한다.
- `/mypage/restock`은 백엔드와 데이터 모델이 없으므로 메뉴와 라우트를 제거한다.
- `/auth/find-email`은 복구 동작이 없으므로 로그인 화면의 링크와 라우트를 제거하고 비밀번호 재설정 흐름만 유지한다.
- `/support/offline/[id]` 라우트가 없으므로 오프라인 목록의 상세보기와 길찾기 링크를 제거한다. 목록에는 가상 매장 데이터라는 점을 명시한다.
- Auth 계정을 생성하지 않는 관리자 사용자 추가 UI와 클라이언트 서비스 메서드는 제거한다.
- 중복된 추천 상품 관리 화면은 실제 내비게이션과 데이터 모델을 기준으로 하나로 통합한다. 유지된 추천 설정은 홈에서 실제로 소비한다.

### 보존 대상

- 인증된 관리자 라우트
- App Router API 프록시
- Firebase Functions 핸들러
- 시드·마이그레이션·이미지 검증 스크립트
- 현재 홈 연결은 없지만 후속 단계에서 연결할 추천 상품 및 정적 콘텐츠 서비스

## 단계 2: 보안·품질 게이트

### 계정 상태

- 사용자 문서가 존재하고 상태가 `active`인 계정만 민감 Function 호출과 사용자 쓰기를 허용한다.
- `inactive`, `banned`, `deleted` 또는 사용자 문서가 없는 계정은 거부한다.
- 관리자 정지·삭제는 서버 Function에서 Auth 비활성화, refresh token 회수와 사용자 문서 상태 변경을 함께 수행한다.
- 관리자 승격은 custom claims 갱신과 refresh token 회수, 필요한 Auth 복구를 마친 뒤 사용자 문서 역할을 마지막에 열고, 강등은 사용자 문서 역할을 먼저 닫은 뒤 claims와 token을 갱신한다.
- 민감 Function은 revoked-token 검증을 사용한다.
- 클라이언트는 권한 변경 후 토큰을 강제 갱신하고 실패 시 로그아웃한다.
- 관리자 판정은 토큰의 admin claim뿐 아니라 사용자 문서의 `role == admin`과 `status == active`를 모두 요구한다. 따라서 기존 관리자 토큰이 남아 있어도 사용자 문서의 역할이 변경되면 Firestore, Storage, Function에서 즉시 거부된다.
- 정지·삭제는 사용자 문서를 먼저 비활성 상태로 바꾼 뒤 Auth 비활성화와 토큰 회수를 수행한다. 권한을 여는 승격·재활성화는 claim/Auth와 기존 token을 먼저 정리하고 사용자 문서를 마지막에 연다. 최종 문서 쓰기의 응답이 모호하면 rollback하지 않고 결과 미확정 오류를 반환한다. 미커밋 상태와 승인된 커밋 상태 모두 동일 요청 재시도로 수렴시킨다.

### Firestore 쓰기 경계

- 일반 사용자와 관리자의 주문 생성·상태 변경·삭제는 Function을 통해서만 수행한다.
- 주문 컬렉션의 클라이언트 직접 update/delete는 차단한다.
- 문의 생성은 `userId == request.auth.uid`, 허용 필드, `status == waiting`, 답변 필드 부재를 강제한다.
- QnA 작성자 정보는 인증 사용자 기준으로 검증하고 임의 시간·상태 필드를 허용하지 않는다.
- 계정 상태를 확인하는 공통 rules helper를 사용한다.
- Firestore와 Storage의 관리자 rules helper는 token claim, 사용자 문서 역할과 활성 상태를 함께 확인한다.

### 공개 AI 상담

- 인증 사용자는 UID, 비회원은 익명 세션과 해시된 네트워크 식별자를 기준으로 제한한다.
- 기본 제한은 분당 10회, 일 100회다.
- 원본 IP 주소는 저장하지 않는다.
- 제한 카운터는 서버에서 원자적으로 증가시키고 초과 시 `429`와 재시도 가능 시점을 반환한다.
- AI 응답에는 `Cache-Control: no-store`를 적용한다.
- AI 호출 실패 시 기존 규칙 기반 답변으로 안전하게 대체한다.

### 운영 의존성과 검증 명령

- `npm audit --omit=dev`에서 확인된 취약 패키지는 호환 가능한 패치 버전으로 올린다.
- 자동 `npm audit fix`로 범위를 넓히지 않고 Next.js와 Firebase 의존성 호환성을 개별 검증한다.
- `test:rules`를 `ci`와 `verify`에 포함한다.
- Functions 빌드가 tracked source를 덮어쓰지 않도록 동기화 명령을 compare-only 검증으로 바꾼다.
- 테스트 import만으로 `.env.local`과 Admin SDK가 초기화되지 않도록 CLI entrypoint와 순수 helper를 분리한다.

### 운영 보조 기능

- 사용자 CSV는 RFC 4180 escaping을 적용한다.
- `=`, `+`, `-`, `@`로 시작하는 셀은 수식으로 실행되지 않도록 중립화한다.
- 관리자 사용자 생성 기능은 신규 초대 시스템을 만들지 않고 제거한다.

## 단계 3: 정책·구매 흐름

### 안내 정책

- UI에서 사용하는 상거래 안내 문구를 타입이 있는 공통 정책 모듈로 모은다.
- 실제 구현이 없는 생일 쿠폰, 등급별 적립, 구매 1% 적립, 간편결제와 당일 출고 문구를 제거한다.
- 가입 혜택은 실제 서버 값인 5,000P로 통일한다.
- 헤더, 장바구니, 체크아웃, 상담 답변과 법적 안내가 같은 정책 표현을 사용한다.
- 실제 결제가 없는 포트폴리오 데모이며 입력 데이터가 Firebase에 저장될 수 있다는 점을 회원가입과 결제 흐름에 명시한다.
- 쿠폰 시드는 실행일 기준 상대 날짜를 사용해 언제 실행해도 유효 쿠폰과 만료 쿠폰을 함께 검증할 수 있게 한다.

### 이벤트 참여 조건

- 이벤트에는 `eligibilityType`을 `none`, `purchase`, `delivered`, `review` 중 하나로 저장한다.
- `purchase`와 `delivered`는 로그인 사용자의 대상 상품 주문과 주문 상태를 확인한다.
- `review`는 배송 완료 주문과 해당 주문 상품의 구매 인증 리뷰를 확인한다.
- 정원, 기간, 중복 참여, 자격과 보상 발급을 하나의 서버 트랜잭션 경계에서 처리한다.
- 기존 이벤트에 자격 값이 없으면 보상 없는 일반 이벤트는 `none`, 리뷰 보상 문구가 있는 이벤트는 `review`로 명시적으로 보정한다.
- 관리자 이벤트 폼은 자격 조건과 실제 지급 가능한 보상 타입만 입력받는다.

### 로그인 후 의도 복원

- 로그인 redirect는 같은 출처의 내부 경로만 허용한다.
- 상품 URL과 선택 옵션, 수량, `cart`·`buy`·`wishlist` 의도는 세션 저장소에 짧게 보관한다.
- 로그인 성공 후 원래 상품으로 돌아와 저장된 의도를 한 번만 복원하고 즉시 삭제한다.
- 세션 데이터가 없거나 상품·옵션이 유효하지 않으면 상품 상세로 안전하게 복귀하고 사용자에게 다시 선택하도록 안내한다.
- 이벤트 참여도 현재 이벤트 URL을 redirect에 포함한다.

### 주문 후 데이터 일관성

- 주문 성공 시 장바구니, 포인트 잔액·내역, 사용자 쿠폰과 주문 목록 캐시를 함께 갱신한다.
- 쿠폰 만료는 `Asia/Seoul` 날짜 단위로 판정해 만료 당일의 클라이언트·서버 결과를 일치시킨다.
- 만료 상태 저장과 오류 응답을 같은 트랜잭션의 rollback에 의존하지 않도록 분리한다.
- 바로구매 체크아웃에서도 사용 가능한 쿠폰을 선택할 수 있게 한다.

## 단계 4: 접근성·사용자 경험

### 폼과 오류

- 공용 `Input`은 `id`, 연결된 `label`, 오류 ID, `aria-invalid`, `aria-describedby`, `required`, `autocomplete`를 지원한다.
- 회원가입과 회원정보 수정은 필드별 오류를 입력에 연결하고 제출 실패 시 첫 오류로 포커스를 이동한다.
- 약관 체크박스 그룹은 `fieldset`과 `legend`를 사용한다.

### 모달·메뉴·채팅

- 안내 팝업은 dialog 의미, 제목 연결, 초기 포커스, 포커스 트랩, Escape 닫기와 트리거 복귀를 제공한다.
- 모바일 메뉴가 열리면 배경을 inert 처리하고 스크롤을 잠그며 닫을 때 메뉴 버튼으로 포커스를 돌린다.
- 채팅창은 토글에 `aria-expanded`와 `aria-controls`, 메시지 목록에 `role=log`, 입력에 명시적 라벨을 제공한다.
- 채팅을 열면 현재 사용 가능한 첫 입력 또는 선택지로 포커스를 이동한다.

### 캐러셀·상품·이벤트

- 메인 캐러셀에 재생·정지 버튼을 제공하고 hover·focus 중 자동 전환을 멈춘다.
- `prefers-reduced-motion`에서는 자동 전환과 불필요한 슬라이드 애니메이션을 중지한다.
- 모바일 페이지 점은 시각 크기와 별개로 44px 터치 영역을 확보한다.
- 상품 카드 컨테이너를 비인터랙티브 요소로 바꾸고 상세 링크와 찜 버튼을 형제로 배치한다.
- 찜 버튼은 상태에 맞는 접근성 이름과 `aria-pressed`를 제공한다.
- 이벤트 목록은 유일한 `h1`, 카드별 텍스트 `h2`와 혜택 설명을 이미지 밖에 제공한다.
- 필터는 `aria-pressed`, 현재 페이지는 `aria-current=page`를 제공한다.

### 실제 동작과 모바일 밀도

- 주문 기간 필터는 선택 기간을 주문일에 적용한다.
- 주문 상태 CSS는 실제 영문 상태 키와 일치시킨다.
- 마이페이지 주문·쿠폰 수는 실제 쿼리 결과를 사용한다.
- 모바일 상품 목록의 장식용 통계는 축소하고 필터는 접을 수 있는 한 영역으로 제공한다.
- 핵심 터치 대상은 최소 44px, 일반 텍스트는 배경 대비 4.5:1 이상을 목표로 한다.
- 마이페이지 하위 화면에는 모바일 대체 내비게이션을 제공한다.

## 단계 5: 데이터·성능·SEO

### 조회 구조

- 루트 Provider에는 인증과 React Query만 유지한다.
- 상품, 이벤트와 쿠폰 데이터는 사용하는 페이지 또는 해당 라우트 레이아웃에서만 조회한다.
- React Query의 공통 query key와 무효화 helper를 사용해 동일 데이터의 중복 요청을 합친다.
- 헤더 카테고리와 상품 목록 카테고리는 같은 캐시를 공유한다.
- 상품 상세와 이벤트 상세에서 메타데이터와 본문이 같은 데이터를 조회할 때 React `cache()`로 요청을 중복 제거한다.
- 데이터 장애는 404로 변환하지 않고 오류 화면과 재시도를 제공한다. 실제 문서 부재만 404로 처리한다.
- 최근 본 상품이나 찜 상품의 개별 조회 실패는 영구 스켈레톤 대신 삭제된 상품·재시도 상태로 표시한다.

### 상품 목록과 검색

- `/products`와 `/categories/[category]`는 공통 상품 목록, 카드, 필터, 정렬과 커서 페이징을 사용한다.
- 카테고리, 정렬, 필터와 페이지 상태는 URL에 반영한다.
- 전체 수를 계산하지 않는 경우 `총 N개` 대신 `현재 페이지 N개`라고 표시한다.
- 검색은 상품명뿐 아니라 카테고리, 브랜드와 한국어 동의어를 정규화해 비교한다.
- 인기 검색어는 결과가 없는 고정 문구 대신 실제 카테고리·검색 매핑과 일치시킨다.

### 홈 큐레이션

- 추천 상품 관리 화면은 하나만 유지한다.
- 유지된 추천 설정의 제목, 설명과 상품 목록을 홈 추천 영역이 직접 사용한다.
- 홈의 MD 콘텐츠는 연결 대상이 있는 경우에만 링크를 렌더링한다.
- 배너의 Firestore 관리 전환은 현재 정적 콘텐츠 서비스가 정상 데이터를 제공할 때만 적용하며, 실패 시 기존 정적 배너를 안전한 fallback으로 사용한다.

### SEO와 이미지

- 루트의 홈페이지 고정 canonical을 제거한다.
- 홈, 상품 목록, 카테고리, 이벤트와 법적 페이지에 각자의 title, description과 canonical을 제공한다.
- 상품 상세에는 Product JSON-LD를 제공한다.
- Open Graph 이미지 MIME은 실제 파일 확장자 또는 응답 메타와 일치시킨다.
- 각 페이지의 최초 LCP 후보 이미지에만 `priority` 또는 `fetchPriority=high`를 적용한다.

## 단계 6: 최종 안정성 검증

### 단계별 검증

- 각 변경은 실패하는 단위 또는 통합 테스트를 먼저 추가한다.
- 관련 테스트, 타입체크와 린트를 통과한 뒤 다음 단계로 진행한다.
- 보안 규칙 변경은 허용·거부 권한 행렬을 Firestore·Storage Emulator에서 검증한다.
- 사용자 인터랙션 변경은 Testing Library에서 키보드와 접근성 상태를 함께 검증한다.

### 최종 명령

1. `npm run typecheck`
2. `npm run lint -- --max-warnings=0`
3. `npm test -- --runInBand`
4. `npx tsc --noEmit --pretty false -p functions/tsconfig.json`
5. `npm run test:rules`
6. `npm run build`
7. `npm audit --omit=dev`

### 브라우저 QA

- 360×800, 390×844, 768×1024, 1440×900에서 홈, 상품 목록, 상품 상세, 검색, 이벤트 목록·상세, 로그인·회원가입, 장바구니·체크아웃, 마이페이지와 404를 확인한다.
- 가로 넘침, 콘솔 오류, LCP 경고, 키보드 포커스, 모바일 메뉴와 모달, 로그인 복귀, 필터·페이지네이션을 확인한다.
- 실제 결제, 배포, 외부 메시지 발송은 수행하지 않는다.

## 완료 조건

- 근거 없이 삭제된 운영 코드가 없다.
- 빈 사용자 메뉴와 끊어진 링크가 남지 않는다.
- 비활성·차단·삭제 계정과 강등된 관리자의 민감 작업이 차단된다.
- 공개 AI 상담의 사용량 제한이 서버에서 적용된다.
- 화면의 혜택·결제·배송·이벤트 안내가 실제 동작과 일치한다.
- 로그인 후 선택한 상품과 의도를 잃지 않는다.
- 주문 완료 직후 포인트·쿠폰·주문 상태가 최신 값으로 보인다.
- 구매 핵심 화면의 폼, 모달, 메뉴, 캐러셀과 상품 카드가 키보드·스크린리더 상태를 제공한다.
- 불필요한 전역 Firestore 조회와 상세 데이터 중복 조회가 제거된다.
- 각 주요 페이지가 정확한 제목, canonical과 구조화 데이터를 제공한다.
- 최종 검증 명령과 브라우저 QA 결과를 사실대로 보고한다.
