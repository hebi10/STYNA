# 체크아웃 배송지 직접 입력 설계

## 목표

회원 프로필에 저장된 배송지가 없어도 주문·결제 화면에서 배송지를 직접 입력해 주문할 수 있게 한다. 사용자가 원하면 입력한 배송지를 회원 프로필에도 저장한다.

## 화면 동작

- 저장된 배송지가 있으면 기존처럼 주소 목록에서 선택할 수 있다.
- 배송 주소 영역에 `새 배송지 입력` 선택지를 함께 제공한다.
- 저장된 배송지가 없으면 `새 배송지 입력`을 기본 선택하고 체크아웃 화면을 계속 표시한다.
- 직접 입력 항목은 배송지명, 받는 분, 연락처, 우편번호, 주소, 상세 주소로 구성한다.
- `입력한 배송지 저장하기` 체크박스는 기본적으로 켜 둔다.
- 저장된 배송지를 선택한 경우에는 저장 체크박스를 표시하지 않는다.
- 기존의 `배송지 등록 필요` 차단 화면은 제거한다.

## 주문 및 저장 흐름

1. 직접 입력 주소를 선택하면 필수 항목을 검증한다.
2. 유효한 직접 입력 주소를 기존 주문 API의 `deliveryAddress`로 전달한다.
3. 주문 생성이 성공한 뒤 저장 체크가 켜져 있으면 `users/{uid}.addresses`에 해당 주소를 추가한다.
4. 저장 체크가 꺼져 있으면 주문에만 주소를 기록한다.
5. 주소 저장 실패는 이미 성공한 주문을 취소하지 않는다. 주문 완료 화면으로 이동하되 저장 실패를 사용자에게 안내한다.

## 데이터 규칙

- 직접 입력 주소에는 클라이언트에서 충돌 가능성이 낮은 고유 ID를 부여한다.
- 새 주소가 첫 저장 주소이면 기본 배송지로 저장한다.
- 기존 기본 배송지가 있으면 새 주소는 일반 배송지로 추가한다.
- 기존 주소 배열을 보존한 채 새 주소만 추가하며 다른 프로필 필드는 변경하지 않는다.
- 서버 주문 API의 기존 배송지 검증과 주문 문서 저장 구조는 그대로 사용한다.

## 오류 처리

- 받는 분, 연락처, 우편번호, 주소가 비어 있으면 주문을 시작하지 않고 항목별 오류를 표시한다.
- 연락처는 현재 회원정보 화면과 같은 국내 휴대전화 형식을 허용한다.
- 프로필 주소 저장 실패는 콘솔에 원인을 남기고 사용자에게 주소가 저장되지 않았음을 알린다.
- 주문 생성 실패 시에는 주소도 저장하지 않는다.

## 테스트 및 검증

- 저장 주소가 없는 사용자도 직접 입력 주소를 만들 수 있는지 단위 테스트한다.
- 첫 저장 주소가 기본 배송지가 되는지, 기존 기본 주소가 있으면 일반 주소로 추가되는지 단위 테스트한다.
- 필수 입력 누락과 연락처 형식 오류를 단위 테스트한다.
- 체크아웃 관련 테스트, 타입체크, 린트를 실행한다.
- 가능하면 브라우저에서 저장 주소 없음, 저장 체크 켜짐/꺼짐, 기존 주소 선택 흐름을 확인한다.

## 비범위

- 외부 주소 검색 서비스 또는 우편번호 API 연동
- 마이페이지 배송지 CRUD 화면 개편
- 주문 API 및 Firestore 보안 정책의 구조 변경
